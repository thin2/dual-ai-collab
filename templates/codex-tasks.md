## 🔖 任务状态说明

| 状态 | 说明 | 负责人 | 下一步 |
|------|------|--------|--------|
| **OPEN** | 待开始 | - | Codex 开始开发 |
| **IN_PROGRESS** | 进行中 | Codex | Codex 完成开发 |
| **DONE** | 已完成（待审计） | Codex | Claude 审计代码 |
| **VERIFIED** | 已验收通过 | Claude | 任务完成 ✅ |
| **REJECTED** | 审计未通过（需修复） | Claude | Codex 修复问题 |
| **BLOCKED** | 阻塞中 | - | 解决阻塞问题 |

---

## 🎯 优先级说明

| 优先级 | 说明 | 示例 | 处理时间 |
|--------|------|------|---------|
| **P1** | 高优先级（紧急且重要） | 核心功能、阻塞性 Bug、安全漏洞 | 立即处理 |
| **P2** | 中优先级（重要但不紧急） | 功能增强、性能优化、代码重构 | 1-3 天内 |
| **P3** | 低优先级（可延后） | 文档完善、UI 优化、技术债务 | 有空时处理 |

## 📐 任务拆分原则

### ✅ 好的任务拆分

```markdown
## 任务 #001: 实现用户登录表单 UI
## 任务 #002: 实现登录表单验证逻辑
## 任务 #003: 实现登录 API 调用
## 任务 #004: 实现 Token 存储和管理
## 任务 #005: 实现登录错误处理
```

**特点**：
- 每个任务独立、可测试
- 任务粒度：2-4 小时
- 明确的验收标准

### ❌ 不好的任务拆分

```markdown
## 任务 #001: 实现整个用户认证系统
```

**问题**：
- 任务过大，难以评估
- 验收标准不明确
- 难以追踪进度

## 🔍 审计标准

Claude 审计代码时会检查以下方面：

### 1. 代码质量 (30%)
- [ ] 代码结构清晰
- [ ] 命名规范
- [ ] 注释适当
- [ ] 无冗余代码

### 2. 设计规范 (20%)
- [ ] 符合项目架构
- [ ] 遵循设计模式
- [ ] 组件解耦合理

### 3. 功能完整性 (25%)
- [ ] 所有验收标准满足
- [ ] 边界情况处理
- [ ] 错误处理完善

### 4. 性能优化 (10%)
- [ ] 无明显性能问题
- [ ] 资源使用合理
- [ ] 响应时间可接受

### 5. 安全性 (10%)
- [ ] 无安全漏洞
- [ ] 输入验证
- [ ] 权限控制

### 6. 可维护性 (5%)
- [ ] 易于理解
- [ ] 易于修改
- [ ] 易于测试

**通过标准**: 总分 ≥ 90 分

## 📅 工作流程

```
1. Claude 创建任务
   ↓
2. Codex 读取任务板，找到最高优先级的 OPEN 任务
   ↓
3. Codex 更新任务状态为 IN_PROGRESS
   ↓
4. Codex 开发代码
   ↓
5. Codex 更新任务状态为 DONE
   ↓
6. Claude 审计代码
   ↓
7. Claude 更新任务状态为 VERIFIED（通过）或 REJECTED（未通过）
   ↓
8. 如果 REJECTED，Codex 修复问题，回到步骤 3
   ↓
9. 如果 VERIFIED，任务完成 ✅
```

## 💡 使用技巧

### Codex 开发时

1. **开始前**：
   ```bash
   # 读取任务板
   cat planning/codex-tasks.md

   # 找到最高优先级的 OPEN 任务
   grep -A 20 "状态\*\*: OPEN" planning/codex-tasks.md | head -n 25
   ```

2. **开发中**：
   - 严格按照任务描述和技术要求开发
   - 确保满足所有验收标准
   - 编写清晰的代码和注释

3. **完成后**：
   - 更新任务状态为 DONE
   - 自测所有验收标准
   - 提交代码

### Claude 审计时

1. **审计前**：
   ```bash
   # 找到所有 DONE 状态的任务
   grep -A 20 "状态\*\*: DONE" planning/codex-tasks.md
   ```

2. **审计中**：
   - 阅读相关文件
   - 检查代码质量
   - 验证验收标准
   - 记录问题和建议

3. **审计后**：
   - 生成审计报告
   - 更新任务状态（VERIFIED 或 REJECTED）
   - 如果 REJECTED，详细说明问题

## 🚨 常见问题

### Q1: 任务太大怎么办？
**A**: 将任务拆分为多个子任务，每个子任务 2-4 小时完成。

### Q2: 任务被阻塞怎么办？
**A**: 更新任务状态为 BLOCKED，在备注中说明阻塞原因，优先解决阻塞问题。

### Q3: 审计不通过怎么办？
**A**: 阅读审计报告，理解问题，修复后重新提交审计。

### Q4: 如何处理紧急 Bug？
**A**: 创建 P1 优先级任务，立即处理，完成后立即审计。

### Q5: 任务依赖关系怎么处理？
**A**: 在任务中标注依赖关系，按依赖顺序执行。
