# Prompt Templates — planning-collab 规划阶段模板

本文件只保留规划阶段的模板，服务于这条链路：
`复杂度判断 -> 访谈 -> 规格生成 -> 任务板 -> 规划审计 -> 用户确认`

设计思路结合了两类稳定实践：
- Anthropic 风格：指令清晰、成功标准明确、用结构化标签分段、鼓励暴露未知项
- OpenAI 风格：稳定角色和长期约束放在上层指令，本轮任务、材料、验收标准放在当前 prompt

## 使用原则

1. 一次调用只做一个阶段目标，不把"审查 + 修改 + 复审"塞进同一轮。
2. 先写验收标准，再写背景材料；先写约束，再写自由发挥空间。
3. 优先给文件路径、任务编号、验收条目，不要把整篇长文贴进 prompt。
4. 强制输出格式，方便 Claude 汇总、决策和回填任务板。
5. 缺信息时要求模型明确指出"缺什么、影响什么"，不要硬猜。

推荐统一使用标签分段：

```text
<task></task>
<context></context>
<constraints></constraints>
<acceptance></acceptance>
<output></output>
```

## 1. 规划阶段

### 1.0 复杂度判断 Prompt

适用对象：Claude
适用场景：正式访谈前，先判断应该走轻流程还是重流程

```text
请先根据用户当前描述判断需求复杂度，不要直接进入默认的重访谈。
<task>
用户需求：[原始描述]
当前项目上下文：[已有信息]
</task>
<output>
严格输出：
## Complexity
SIMPLE / MEDIUM / COMPLEX
## Reason
[1-3 句，解释为什么]
## Interview Depth
[2-3 轮 / 4-6 轮 / 6-10 轮]
## Must Clarify Next
1. [问题]
2. [问题]
</output>
```

### 1.1 规划文档主审 Prompt

适用对象：Claude
适用场景：需求文档和任务板生成后，开发前的第一道质量门
输入项：用户目标、需求文档路径、任务板路径、审计重点

```text
你是资深产品技术规划审查员。请判断当前规划文档是否已经达到"可以开始开发"的标准。
<task>
用户目标：[一句话重述]
需求文档：[路径]
任务板：[路径]
项目路径：$(pwd)
本轮重点：[范围边界 / 依赖 / 验收标准 / 风险]
</task>
<requirements>
重点检查：
1. 是否覆盖用户原始目标，没有明显遗漏或误解
2. 任务拆分是否可执行，边界是否清晰
3. 依赖顺序是否合理
4. 验收标准是否可验证，不是空泛描述
5. 风险、边界情况、回滚思路是否有记录
6. 是否存在过度设计
</requirements>
<constraints>
- 先基于文档和项目事实判断，不要脑补需求
- 缺信息时写清"缺什么、影响什么"
- 以"能否开始开发"为判断标准
</constraints>
<output>
严格输出：
## 结论
READY / NOT_READY
## Blocking Issues
1. [没有则写"无"]
## Non-blocking Suggestions
1. [没有则写"无"]
## Missing Decisions
1. [没有则写"无"]
## Coverage Check
- 目标覆盖：[充分 / 部分 / 不足] - [说明]
- 任务拆分：[充分 / 部分 / 不足] - [说明]
- 验收标准：[充分 / 部分 / 不足] - [说明]
- 风险记录：[充分 / 部分 / 不足] - [说明]
## Recommendation
[直接给用户确认 / 先补规划再确认]
</output>
```

### 1.2 规划补漏 Prompt

适用对象：Codex / 其他 AI
适用场景：Claude 已完成主审，只需要补查遗漏
输入项：需求文档路径、任务板路径、Claude 主审摘要、补查方向

```text
你是规划文档补漏审查员。Claude 已完成主审，你只查漏补缺，不要重复完整审计。
<task>
需求文档：[路径]
任务板：[路径]
Claude 主审摘要：[READY/NOT_READY、Blocking Issues、Missing Decisions]
本轮只补查：[依赖遗漏 / 验收标准可测性 / 潜在漏任务 / 风险缺口]
</task>
<constraints>
- 只写新增发现，不重复 Claude 已指出的问题
- 没有新增发现就明确写"无新增问题"
- 每个问题都要说明影响哪个后续任务或阶段
</constraints>
<output>
严格输出：
## 新增问题
1. [问题] - [影响]
## 新增建议
1. [建议] - [收益]
## 是否阻塞开发
YES / NO
## 一句话总结
[最多 2 句]
</output>
```

### 1.3 用户确认规划 Prompt

适用对象：Claude -> 用户
适用场景：规划审计完成后，请用户亲自确认再开发
输入项：需求文档路径、任务板路径、审计摘要、总任务数、预计工时

```text
规划文档已经准备好，请你先确认，再决定是否进入开发。
规划文件：
- 需求文档：[路径]
- 任务板：[路径]
审计摘要：
- 主结论：[READY / NOT_READY]
- 必改问题：[1-3 条，没有则写"无"]
- 待你确认的决策：[1-3 条，没有则写"无"]
执行概览：
- 总任务数：[X]
- 预计总工时：[X 小时]
请直接回复以下之一：
1. 开始开发
2. 修改规划
3. 取消
```

## 2. 异常升级

### 2.1 方案分叉确认 Prompt

适用对象：Claude -> 用户
适用场景：存在多个可行方案，且代价或影响差异明显

```text
当前有两个可行方案，需要你拍板后再继续。

方案 A：[名称]
- 优点：[1-2 条]
- 缺点：[1-2 条]
- 成本：[工时 / 风险]

方案 B：[名称]
- 优点：[1-2 条]
- 缺点：[1-2 条]
- 成本：[工时 / 风险]

我的建议：[推荐方案]，因为 [一句话理由]。
请直接回复你选择的方案。
```

### 2.2 破坏性修改确认 Prompt

适用对象：Claude -> 用户
适用场景：涉及公共接口变更、数据迁移、删除关键文件等

```text
这个任务涉及破坏性修改，继续前需要你确认。

变更内容：
- [变更 1]
- [变更 2]

影响范围：
- [模块 / 接口 / 数据]

风险：
- [风险 1]
- [风险 2]

回滚方式：
- [如何撤回]

请回复：继续 或 取消。
```

## 3. 使用建议

推荐默认顺序：
1. `1.0` 复杂度判断
2. 访谈（按复杂度分级，Claude 主导）
3. 生成规格文档 + 任务板
4. `1.1` 规划文档主审
5. `1.2` 规划补漏（仅在需要时）
6. `1.3` 用户确认规划

避免这些写法：
- 不要在同一个 prompt 里同时要求"审查 + 修改 + 复审"
- 不要只写"请帮我规划这个功能"，却不给目标和边界
- 不要把"风格要优雅、代码要高级"这类空话当约束
- 不要把补充审查做成对主审的重复劳动
