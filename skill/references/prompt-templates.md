# Prompt Templates — dual-ai-collab 阶段化模板

本文件只保留当前流程最有价值的模板，服务于这条简化链路：
`规划审计 -> 用户确认 -> 开发执行 -> 最终审查 -> 返工修复（按需）`

设计思路结合了两类稳定实践：
- Anthropic 风格：指令清晰、成功标准明确、用结构化标签分段、鼓励暴露未知项
- OpenAI 风格：稳定角色和长期约束放在上层指令，本轮任务、材料、验收标准放在当前 prompt

## 使用原则

1. 一次调用只做一个阶段目标，不把“审查 + 修改 + 复审”塞进同一轮。
2. 先写验收标准，再写背景材料；先写约束，再写自由发挥空间。
3. 优先给文件路径、任务编号、验收条目，不要把整篇长文贴进 prompt。
4. 强制输出格式，方便 Claude 汇总、决策和回填任务板。
5. 缺信息时要求模型明确指出“缺什么、影响什么”，不要硬猜。
6. Claude 主审后，如果让 Codex 补漏，只查增量问题，不重复整份审计。

推荐统一使用标签分段：

```text
<task></task>
<context></context>
<constraints></constraints>
<acceptance></acceptance>
<output></output>
```

## 1. 规划阶段

### 1.0 复杂度判断 Prompt

适用对象：Claude  
适用场景：正式访谈前，先判断应该走轻流程还是重流程

```text
请先根据用户当前描述判断需求复杂度，不要直接进入默认的重访谈。
<task>
用户需求：[原始描述]
当前项目上下文：[已有信息]
</task>
<output>
严格输出：
## Complexity
SIMPLE / MEDIUM / COMPLEX
## Reason
[1-3 句，解释为什么]
## Interview Depth
[2-3 轮 / 4-6 轮 / 6-10 轮]
## Must Clarify Next
1. [问题]
2. [问题]
</output>
```

### 1.1 规划文档主审 Prompt

适用对象：Claude  
适用场景：需求文档和任务板生成后，开发前的第一道质量门  
输入项：用户目标、需求文档路径、任务板路径、审计重点

```text
你是资深产品技术规划审查员。请判断当前规划文档是否已经达到“可以开始开发”的标准。
<task>
用户目标：[一句话重述]
需求文档：[路径]
任务板：[路径]
项目路径：$(pwd)
本轮重点：[范围边界 / 依赖 / 验收标准 / 风险]
</task>
<requirements>
重点检查：
1. 是否覆盖用户原始目标，没有明显遗漏或误解
2. 任务拆分是否可执行，边界是否清晰
3. 依赖顺序是否合理
4. 验收标准是否可验证，不是空泛描述
5. 风险、边界情况、回滚思路是否有记录
6. 是否存在过度设计
</requirements>
<constraints>
- 先基于文档和项目事实判断，不要脑补需求
- 缺信息时写清“缺什么、影响什么”
- 以“能否开始开发”为判断标准
</constraints>
<output>
严格输出：
## 结论
READY / NOT_READY
## Blocking Issues
1. [没有则写“无”]
## Non-blocking Suggestions
1. [没有则写“无”]
## Missing Decisions
1. [没有则写“无”]
## Coverage Check
- 目标覆盖：[充分 / 部分 / 不足] - [说明]
- 任务拆分：[充分 / 部分 / 不足] - [说明]
- 验收标准：[充分 / 部分 / 不足] - [说明]
- 风险记录：[充分 / 部分 / 不足] - [说明]
## Recommendation
[直接给用户确认 / 先补规划再确认]
</output>
```

### 1.2 规划补漏 Prompt

适用对象：Codex  
适用场景：Claude 已完成主审，只需要 Codex 补查遗漏  
输入项：需求文档路径、任务板路径、Claude 主审摘要、补查方向

```text
你是规划文档补漏审查员。Claude 已完成主审，你只查漏补缺，不要重复完整审计。
<task>
需求文档：[路径]
任务板：[路径]
Claude 主审摘要：[READY/NOT_READY、Blocking Issues、Missing Decisions]
本轮只补查：[依赖遗漏 / 验收标准可测性 / 潜在漏任务 / 风险缺口]
</task>
<constraints>
- 只写新增发现，不重复 Claude 已指出的问题
- 没有新增发现就明确写“无新增问题”
- 每个问题都要说明影响哪个后续任务或阶段
</constraints>
<output>
严格输出：
## 新增问题
1. [问题] - [影响]
## 新增建议
1. [建议] - [收益]
## 是否阻塞开发
YES / NO
## 一句话总结
[最多 2 句]
</output>
```

### 1.3 用户确认规划 Prompt

适用对象：Claude -> 用户  
适用场景：规划审计完成后，请用户亲自确认再开发  
输入项：需求文档路径、任务板路径、审计摘要、总任务数、预计工时

```text
规划文档已经准备好，请你先确认，再决定是否进入开发。
规划文件：
- 需求文档：[路径]
- 任务板：[路径]
审计摘要：
- 主结论：[READY / NOT_READY]
- 必改问题：[1-3 条，没有则写“无”]
- 待你确认的决策：[1-3 条，没有则写“无”]
执行概览：
- 总任务数：[X]
- 预计总工时：[X 小时]
请直接回复以下之一：
1. 开始开发
2. 修改规划
3. 取消
```

## 2. 开发阶段

### 2.1 通用开发任务 Prompt

适用对象：Codex  
适用场景：后端、脚本、数据处理、通用逻辑开发  
输入项：任务编号、标题、目标、相关文件、约束、验收标准、验证命令、环境激活命令

```text
[可选：在执行前先运行：source $(conda info --base)/etc/profile.d/conda.sh && conda activate [环境名]]

你是专业开发工程师，请在当前仓库中完成一个明确任务。
<task>
任务编号：#[XXX]
任务标题：[标题]
任务目标：[一句话说明交付物]
</task>
<context>
相关文件：
- [路径 1]
- [路径 2]
可参考实现：
- [路径 3，可选]
当前约束：
- [如“保持现有 CLI 参数不变”]
- [如“不要改公共接口签名”]
</context>
<acceptance>
- [ ] [标准 1]
- [ ] [标准 2]
- [ ] [标准 3]
</acceptance>
<constraints>
- 先阅读相关文件，再开始修改
- 只改完成此任务所需的文件
- 不要顺手做无关重构
- 如任务描述与代码现实冲突，优先按代码现实处理并在结果中说明
</constraints>
<verification>
完成后尽量自行验证：
- [命令 1]
- [命令 2]
</verification>
<output>
严格输出：
## Result
DONE / BLOCKED
## Files Changed
- [文件路径]
## Acceptance Check
- [标准 1]：PASS / FAIL - [说明]
- [标准 2]：PASS / FAIL - [说明]
## Verification
- [命令] - [结果]
## Notes
- [风险 / 折中 / 未完成项；没有写“无”]
</output>
```

### 2.2 前端 / UI 开发 Prompt

适用对象：Codex  
适用场景：页面、组件、样式、交互开发；前提是 Claude 已生成设计规范  
输入项：任务编号、标题、设计方案路径、相关文件、技术栈、验收标准

前置规则分级：

- 新页面 / 新组件体系 / 新视觉方向 / 复杂交互：先调用 `ui-ux-pro-max`
- 小型样式修复 / 现有组件微调 / 低风险 UI bug：可直接实现，必要时只做轻量设计说明

```text
你是专业前端开发工程师，请严格按设计规范完成实现。
<task>
任务编号：#[XXX]
任务标题：[标题]
设计方案：[路径]
</task>
<context>
技术栈：[如 React + TypeScript + Tailwind]
相关文件：
- [路径 1]
- [路径 2]
</context>
<requirements>
必须做到：
- 视觉风格与设计方案一致
- 适配 mobile / tablet / desktop
- 保持语义化结构和基本无障碍能力
- 尽量复用现有组件和样式体系
</requirements>
<constraints>
- 不要擅自改设计方向
- 不要新增 UI 依赖，除非任务明确要求
- 不要为了“更炫”而偏离当前产品风格
</constraints>
<acceptance>
- [ ] 符合设计方案
- [ ] 响应式正常
- [ ] 交互状态完整（loading / empty / error / hover / disabled）
- [ ] 无明显可访问性问题
</acceptance>
<output>
严格输出：
## Result
DONE / BLOCKED
## Files Changed
- [文件路径]
## Design Fidelity
- 一致部分：[列表]
- 偏差部分：[列表；没有则写“无”]
## Acceptance Check
- [标准]：PASS / FAIL - [说明]
## Notes
- [剩余风险或需要复查的点；没有写“无”]
</output>
```

### 2.3 返工修复 Prompt

适用对象：Codex  
适用场景：审查退回后，按问题清单逐条修复  
输入项：任务编号、修复轮次、问题列表、相关文件、必须保持不变的边界

```text
你需要修复代码审查中发现的问题。本轮目标是逐条对应修复，不是重做整个任务。
<task>
任务编号：#[XXX]
修复轮次：第 [N] 轮
相关文件：
- [路径 1]
- [路径 2]
</task>
<findings>
1. [严重程度：高/中/低] [问题描述]
2. [严重程度：高/中/低] [问题描述]
</findings>
<constraints>
- 必须逐条处理问题
- 未被指出的区域不要大改
- 不要通过删除功能来“修复”问题
- 高严重问题优先
</constraints>
<acceptance>
- 每个问题都有对应修复动作
- 没有引入明显回归
- 原任务核心验收标准仍成立
</acceptance>
<output>
严格输出：
## Result
DONE / BLOCKED
## Fix Mapping
1. 问题：[原问题]
   修复：[做了什么]
   文件：[文件路径]
## Regression Risk
- [没有写“无”]
## Verification
- [命令和结果；没有则说明原因]
</output>
```

## 3. 审查阶段

### 3.1 规范符合性审查 Prompt

适用对象：superpowers `spec-reviewer` / Codex / Claude  
适用场景：第一轮审查，只判断“是否按任务描述和验收标准做了”  
输入项：任务编号、标题、相关文件、验收标准、任务原始目标

```text
你是规范符合性审查员。你的职责不是评价代码风格，而是判断实现是否真的完成了任务目标。
<task>
任务编号：#[XXX]
任务标题：[标题]
任务目标：[一句话交付目标]
相关文件：
- [路径 1]
- [路径 2]
</task>
<acceptance>
- [ ] [标准 1]
- [ ] [标准 2]
- [ ] [标准 3]
</acceptance>
<focus>
只回答这些问题：
1. 是否完成了任务目标
2. 验收标准是否逐项覆盖
3. 是否有遗漏实现或偏题实现
4. 是否存在需要用户补充确认的模糊点
</focus>
<constraints>
- 不要讨论编码风格偏好
- 不要把“可以优化”当成 FAIL
- 没有证据就写 UNCLEAR，不要脑补
</constraints>
<output>
严格输出：
## Verdict
PASS / FAIL / UNCLEAR
## Blocking Issues
1. [没有则写“无”]
## Missing Coverage
1. [没有则写“无”]
## Need User Confirmation
1. [没有则写“无”]
## Acceptance Review
- [标准 1]：PASS / FAIL / UNCLEAR - [说明]
- [标准 2]：PASS / FAIL / UNCLEAR - [说明]
</output>
```

### 3.2 代码质量审查 Prompt

适用对象：superpowers `code-quality-reviewer` / Codex / Claude  
适用场景：第二轮审查，在规范符合性通过后判断实现质量  
输入项：任务编号、标题、相关文件、验收标准、风险关注点

```text
你是资深代码质量审查员。请对以下任务做第二轮审查，重点找结构性问题、回归风险和缺失验证。
<task>
任务编号：#[XXX]
任务标题：[标题]
相关文件：
- [路径 1]
- [路径 2]
</task>
<acceptance>
- [ ] [标准 1]
- [ ] [标准 2]
- [ ] [标准 3]
</acceptance>
<focus>
重点关注：
- 功能正确性
- 边界情况
- 回归风险
- 测试或验证缺口
- 安全与性能问题
</focus>
<constraints>
- 发现问题时优先给出可定位证据
- 没有明显问题就明确写“未发现阻塞问题”
- 不要输出泛泛而谈的风格建议
- 不要重复第一轮已经确认通过的规范符合性内容
</constraints>
<output>
严格输出：
## Verdict
PASS / FAIL
## Findings
1. [严重程度：高/中/低] [文件:行号] [问题描述] - [影响]
## Acceptance Review
- [标准 1]：PASS / FAIL / UNCLEAR - [说明]
- [标准 2]：PASS / FAIL / UNCLEAR - [说明]
## Test Gaps
1. [没有则写“无”]
## Recommendation
[通过 / 退回修复 / 需人工确认]
</output>
```

### 3.3 Claude 最终判定模板

适用对象：Claude  
适用场景：读取两轮审查结果后做最终决策

```text
读取以下输入后做最终判定：
1. 任务原始目标
2. 验收标准
3. 第一轮规范符合性审查结果
4. 第二轮代码质量审查结果
5. 实际代码改动

按以下顺序判断：
1. 是否存在阻塞交付的问题
2. 是否有未覆盖的高风险边界
3. 是否缺少关键验证导致无法确认正确性
4. 是否需要用户补充决策

输出：
- 最终结论：VERIFIED / REJECTED / NEED_USER_INPUT
- 核心理由：1-3 条
- 下一步动作：继续 / 修复 / 询问用户
```

## 5. 使用建议

推荐默认顺序：
1. `1.1` 规划文档主审
2. `1.2` 规划补漏（仅在需要时）
3. `1.3` 用户确认规划
4. `2.1` 或 `2.2` 开发执行
5. `3.1` 最终代码审查
6. `2.3` 返工修复（仅在 FAIL 时）

避免这些写法：
- 不要在同一个 prompt 里同时要求“先审查，再修改，再复审”
- 不要只写“请帮我实现这个功能”，却不给验收标准和修改边界
- 不要把“风格要优雅、代码要高级”这类空话当约束
- 不要把补充审查做成对主审的重复劳动
