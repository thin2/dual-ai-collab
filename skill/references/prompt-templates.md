# Prompt Templates — dual-ai-collab 提示词模板

本文件收录 dual-ai-collab 各阶段可直接复用的提示词模板。所有模板均基于 `codex exec --full-auto` 调用方式设计。

---

## 使用原则

1. **输入项必须填满**：模板中的 `[占位符]` 必须替换为真实内容，不允许留空或写"略"。
2. **一次一任务**：每个 Codex 调用只处理一个明确任务，不要塞多个目标。
3. **约束先于自由**：先写约束（不能碰什么、不许改什么），再写期望。
4. **输出格式强制**：要求 Codex 按指定格式输出，便于 Claude 解析和决策。
5. **环境信息前置**：如有虚拟环境，在 prompt 开头加激活命令。

## 1. 上下文收集 Prompt

### 1.1 结构化快速扫描

**适用场景**：任务执行前，快速了解项目现状和相关代码。

**输入项**：
- 功能名称
- 预期涉及的目录/模块

**Prompt 模板**：
```
你是代码分析专家。请对以下项目进行结构化扫描：

【扫描目标】：[功能名称]
【项目目录】：$(pwd)
【关注模块】：[目录列表，如 src/auth、src/models]

请输出以下结构化报告：

## 1. 定位
- 功能所在模块/文件路径
- 入口函数/类

## 2. 现状
- 当前实现方式（1-2 个相似案例）
- 现有代码行数估算

## 3. 技术栈
- 语言/框架/主要依赖版本
- 构建工具和测试框架

## 4. 测试覆盖
- 相关测试文件路径
- 现有测试用例数量

## 5. 观察与风险
- 发现的异常或潜在问题
- 信息不足需要深挖的方向
- 与其他模块的耦合点

只输出事实和证据，不做猜测。
```

### 1.2 针对性深挖

**适用场景**：快速扫描后，对某个高优先级疑问进行深入分析。

**输入项**：
- 具体疑问
- 相关文件路径

**Prompt 模板**：
```
你是代码分析专家。请针对以下疑问进行深入分析：

【疑问】：[具体疑问，如"并发写入如何处理？"]
【相关文件】：[文件路径列表]

要求：
1. 聚焦这一个疑问，不要发散
2. 提供代码片段作为证据
3. 给出 2-3 个可选方案及优劣对比

输出格式：
## 疑问：[重述疑问]
## 代码证据
- [文件:行号] [代码片段]
## 分析结论
[结论]
## 可选方案
1. [方案A] — 优点/缺点
2. [方案B] — 优点/缺点
```

### 1.3 充分性检查

**适用场景**：Claude 在进入任务规划前的自检清单。此模板由 Claude 自行执行，不需要调用 Codex。

**检查清单**：
```
在进入任务规划前，确认以下四项全部打勾：

□ 接口契约：我能定义清晰的输入/输出/异常吗？
□ 技术选型：我理解当前实现的选型理由吗？
□ 风险识别：我知道主要风险点（并发、边界、性能）吗？
□ 验证方式：我知道怎么验证实现是否正确吗？

全部打勾 → 进入任务规划
有未打勾 → 补充一次针对性深挖（1.2）
```

## 2. 任务执行 Prompt

### 2.1 通用实现任务

**适用场景**：后端逻辑、API 开发、数据处理等非前端任务。

**输入项**：
- 任务编号和标题
- 任务描述
- 技术要求
- 验收标准
- 相关文件
- 约束条件

**Prompt 模板**：
```
你是专业开发工程师，执行以下任务：

## 任务
- 编号：#[XXX]
- 标题：[任务标题]
- 描述：[详细描述]

## 技术栈
- 语言/框架：[如 Python 3.11 + FastAPI]
- 参考文件：[现有代码路径，用于对齐风格]

## 技术要求
- [要求 1]
- [要求 2]

## 约束
- 不允许修改：[不能碰的文件或接口]
- 不允许新增依赖：除非明确说明
- 代码风格：与 [参考文件] 保持一致

## 验收标准
- [ ] [标准 1]
- [ ] [标准 2]
- [ ] [标准 3]

完成后列出你修改/创建的所有文件路径。
```

### 2.2 前端/UI 任务

**适用场景**：页面、组件、样式等前端实现任务。需要先由 Claude 调用 `/ui-ux-pro-max` 生成设计方案。

**输入项**：
- 任务编号和标题
- 设计方案内容（来自 ui-ux-pro-max）
- 技术栈
- 验收标准

**Prompt 模板**：
```
你是专业前端开发工程师，执行以下任务：

## 任务
- 编号：#[XXX]
- 标题：[任务标题]

## 设计规范（必须严格遵循）
[粘贴 ui-ux-pro-max 生成的设计方案，包含配色、布局、组件结构、交互规范]

## 技术栈
- 框架：[如 React 18 + TypeScript]
- UI 库：[如 Tailwind CSS + shadcn/ui]
- 参考组件：[现有组件路径，用于对齐风格]

## 约束
- 必须响应式：适配 mobile / tablet / desktop
- 必须无障碍：语义化 HTML，ARIA 标签，键盘导航
- 不允许内联样式
- 不允许新增 UI 依赖

## 验收标准
- [ ] UI 与设计方案一致
- [ ] 响应式适配正常
- [ ] 无障碍检查通过
- [ ] [其他标准]

完成后列出你修改/创建的所有文件路径。
```

### 2.3 修复型任务

**适用场景**：审计 REJECTED 后，Codex 根据审计意见逐条修复。

**输入项**：
- 任务编号
- 审计报告内容（Codex + Claude 双方意见）
- 当前修复轮次

**Prompt 模板**：
```
你需要修复代码审查中发现的问题：

## 任务
- 编号：#[XXX]
- 修复轮次：第 [N]/3 轮
- 相关文件：[文件列表]

## 审计意见（必须逐条修复）

### Codex 审查发现
1. [问题1] — 严重程度：[高/中/低]
2. [问题2] — 严重程度：[高/中/低]

### Claude 审查发现
1. [问题1] — 严重程度：[高/中/低]
2. [问题2] — 严重程度：[高/中/低]

## 修复要求
1. 逐条对应修复，不允许笼统处理
2. 不允许引入新问题
3. 保持代码风格一致
4. 高严重程度问题优先修复

## 输出格式
对每个问题，说明：
- 问题：[原始问题]
- 修复方式：[你做了什么]
- 修改文件：[文件:行号]
```

## 3. 审查 Prompt

### 3.1 Codex 初审

**适用场景**：任务完成（DONE）后，Codex 进行第一轮代码审查。

**输入项**：
- 任务标题和描述
- 相关文件列表
- 验收标准

**Prompt 模板**：
```
你是资深代码审查员，审查以下任务的实现：

## 任务信息
- 任务：#[XXX] [任务标题]
- 描述：[任务描述]
- 相关文件：[文件列表]

## 验收标准
- [ ] [标准 1]
- [ ] [标准 2]

## 审查维度（逐项检查）

1. **功能符合性**：是否满足所有验收标准？
2. **代码质量**：可读性、命名、函数长度、缩进层级（>3 层需标记）
3. **安全性**：注入、XSS、敏感信息泄露、权限控制
4. **性能**：内存泄漏、N+1 查询、不必要的循环、阻塞操作
5. **边界处理**：空值、超大输入、并发、异常路径

## 输出格式（严格遵循）

### 评分：[0-100]
### 结论：PASS / FAIL
### 验收标准核验
- [标准1]：✅ 通过 / ❌ 未通过 — [说明]
### 发现的问题
1. [严重程度:高/中/低] [文件:行号] [问题描述]
2. ...
### 改进建议
1. [建议]
2. ...
```

### 3.2 Claude 终审决策模板

**适用场景**：Claude 读取 Codex 审查报告后做出最终判定。此模板由 Claude 自行执行，不调用 Codex。

**决策规则**：
```
读取以下输入后做出判定：

输入：
1. Codex 审查报告（评分 + 问题列表）
2. 需求规范文档（验收标准）
3. 实际代码

决策矩阵：
┌─────────────────┬──────────────────────┐
│ Codex 评分 ≥ 85  │ 且无高严重问题       │ → VERIFIED
│ Codex 评分 70-84 │ 或有中严重问题       │ → 人工审阅后决策
│ Codex 评分 < 70  │ 或有高严重问题       │ → REJECTED
└─────────────────┴──────────────────────┘

输出：
- Claude 审查状态：VERIFIED / REJECTED
- Claude 评分：[0-100]
- 综合评分：(Codex评分 × 0.4 + Claude评分 × 0.6)
- 补充意见：[Codex 未发现但 Claude 发现的问题]
```

## 4. 升级与确认 Prompt

### 4.1 多方案分叉

**适用场景**：Codex 上下文收集发现多个技术方案，需要 Claude 向用户确认。

**Prompt 模板（Claude → 用户）**：
```
在分析 [功能名称] 时发现多种实现方案，需要你的决策：

方案 A：[方案名称]
- 优点：[列举]
- 缺点：[列举]
- 工时：约 X 小时

方案 B：[方案名称]
- 优点：[列举]
- 缺点：[列举]
- 工时：约 X 小时

建议：[推荐方案及理由]
```

### 4.2 破坏性修改确认

**适用场景**：任务涉及修改公共 API、删除文件、变更数据库 schema 等。

**Prompt 模板（Claude → 用户）**：
```
⚠️ 任务 #[XXX] 涉及破坏性修改，需要确认：

变更内容：
- [具体变更 1]
- [具体变更 2]

影响范围：
- 受影响的文件/模块：[列举]
- 受影响的 API/接口：[列举]

回滚方案：
- [如何撤回这些修改]

是否继续执行？
```

### 4.3 连续失败升级

**适用场景**：Codex 连续 3 次执行同一任务失败或卡死。

**Prompt 模板（Claude → 用户）**：
```
⚠️ 任务 #[XXX] 连续 [N] 次执行失败，需要人工介入：

任务描述：[标题]

失败记录：
- 第 1 次：[失败原因/错误信息]
- 第 2 次：[失败原因/错误信息]
- 第 3 次：[失败原因/错误信息]

可能原因：
- [分析可能的根本原因]

建议选项：
1. 简化任务范围后重试
2. Claude 直接编写（跳过 Codex）
3. 跳过此任务继续下一个
4. 暂停流程，等待人工修复
```

### 4.4 信息不足暂停

**适用场景**：充分性检查未通过，且深挖 3 次后仍然信息不足。

**Prompt 模板（Claude → 用户）**：
```
⚠️ 在规划 [功能名称] 时，以下信息仍然不足：

缺失信息：
- [具体缺失 1]：影响 [哪个任务/决策]
- [具体缺失 2]：影响 [哪个任务/决策]

已尝试的收集方式：
- [扫描/深挖了什么，为什么没找到]

需要你提供：
- [你需要告诉我什么]
```

## 5. 模板使用建议

### 选择模板的决策流

```
收到任务 →
  ├─ 新项目/不熟悉 → 1.1 结构化扫描 → 1.3 充分性检查
  ├─ 有疑问 → 1.2 针对性深挖（≤3 次）
  ├─ 前端任务 → 先调 ui-ux-pro-max → 2.2 前端模板
  ├─ 后端/通用 → 2.1 通用模板
  ├─ 审计阶段 → 3.1 Codex 初审 → 3.2 Claude 终审
  ├─ 修复阶段 → 2.3 修复型任务
  └─ 遇到阻塞 → 4.x 升级确认模板
```

### 环境信息注入

如果项目有虚拟环境，在所有 Codex Prompt 前面加：
```
在执行前先运行：source $(conda info --base)/etc/profile.d/conda.sh && conda activate [项目名]
```

### Codex 调用注意事项

- 使用 `codex exec -C "$(pwd)" --full-auto` 调用
- 使用 `run_in_background: true` 避免阻塞 Claude
- 日志输出到 `.dual-ai-collab/logs/task-XXX.log`
- PID 记录到 `.dual-ai-collab/logs/task-XXX.pid`

### 未来 MCP 升级

当配置了 Codex MCP Server 后，可将 `codex exec` 替换为：
```javascript
mcp__codex__codex({
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "<上述模板内容>"
})
```
MCP 调用支持 `conversationId` 连续对话，适合多轮修复场景。
